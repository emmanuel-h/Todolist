# Multiple Todo Lists

## What it does
The home screen shows all todo lists; the user can create a new list via a FAB dialog, delete a list with a long-press confirmation, and navigate into a list by tapping it.

## Architecture
- **Layers**: domain, presentation, ui
- **Key types**:
  - `TodoList` — immutable domain model (`id`, `name`)
  - `TodoListsState` — sealed class: `Empty` | `Content(lists: List<TodoList>)`
  - `TodoListsViewModel` — `state: TodoListsState` computed property (recomputed on every access, no StateFlow); methods `createList(name)`, `deleteList(todoListId)`
  - `CreateTodoListUseCase` — validates name is non-blank, generates UUID id
  - `DeleteTodoListUseCase` — deletes all items first via `todoRepository.deleteAllByListId`, then deletes the list
  - `GetTodoListsUseCase` — returns all lists from the repository
- **Async contract**: synchronous; `state` is a plain computed property re-evaluated on each read

## Invariants & contracts
- `DeleteTodoListUseCase` must delete items before the list; Room's FK cascade is a safety net but domain logic must not rely on it exclusively.
- `TodoList.id` is generated by `CreateTodoListUseCase` via `UUID.randomUUID()`; never supply an id from the UI.
- `TodoListsActivity.onResume()` calls `refreshLists()` so the list stays accurate after back-navigation from `TodoListActivity`.
- `TodoListsViewModel.state` is a computed property — callers must not cache its value between mutations.

## UI
- **Screen(s)**: `TodoListsActivity` (launcher)
- **Layout file(s)**: `res/layout/activity_todo_lists.xml`, `res/layout/item_todo_list.xml`, `res/layout/dialog_create_list.xml`
- **Design decisions**: Delete is intentionally on long-press rather than a swipe to reduce accidental deletions; a confirmation dialog is shown before any list is removed.
